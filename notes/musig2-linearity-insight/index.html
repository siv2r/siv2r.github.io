musig2-linearity-insight

Wed Apr 01 2026

MuSig2 has two things to get right: induce linearity in Schnorr signatures, and preserve the already existing relation c=f(x,k)c = f(x, k)c=f(x,k). If we can make partial signatures linear, summing them gives a valid single Schnorr signature, indistinguishable from a normal one.

Schnorr signatures are partially linear

A Schnorr signature resembles the function f(x,y,z)=x+y⋅zf(x, y, z) = x + y \cdot zf(x,y,z)=x+y⋅z. This function is not linear overall (with respect to all xxx, yyy, zzz), but it is linear with respect to xxx and yyy when zzz is constant, and with respect to xxx and zzz when yyy is constant.

The actual signature equation is s=k+c⋅xs = k + c \cdot xs=k+c⋅x. By keeping ccc constant across multiple partial signatures, we induce linearity. Summing the partial signatures then produces a valid Schnorr signature.

How to keep c constant

The challenge value is computed as c=Hnon(R∥P∥msg)c = H_{\text{non}}(R \| P \| \text{msg})c=Hnon​(R∥P∥msg). To make ccc constant across signers, we need RRR, PPP, and msg\text{msg}msg to all be the same for every partial signature.

Preserving the relation between c, k, and x

In f(x,y,z)=x+y⋅zf(x, y, z) = x + y \cdot zf(x,y,z)=x+y⋅z, all three variables are independent. But in Schnorr signatures they are not. The challenge ccc depends on kkk and xxx through the relation:

c=Hnon(gk∥gx∥msg)c = H_{\text{non}}(g^k \| g^x \| \text{msg})c=Hnon​(gk∥gx∥msg)

We also need to preserve this relation when combining signatures. For nnn signers:

R=gk1+k2+⋯+knR = g^{k_1 + k_2 + \cdots + k_n}R=gk1​+k2​+⋯+kn​ P=gx1+x2+⋯+xnP = g^{x_1 + x_2 + \cdots + x_n}P=gx1​+x2​+⋯+xn​

The aggregate RRR and PPP must correspond to the sums of individual nonces and secret keys respectively. Then ccc is consistent across all partial signatures, and their sum is valid under the aggregate key.

Why aggregate coefficients exist

The aggregate public key depends on the order of the individual public keys because of the aggregate coefficient ai=Hagg(L,Xi)a_i = H_{\text{agg}}(L, X_i)ai​=Hagg​(L,Xi​). If the order changes, aia_iai​ changes, so the aggregate key changes. Without these coefficients, a malicious signer could pick their public key as a function of honest signers' keys to cancel them out. That's the rogue key attack, and the aggregate coefficients are what block it.

Open question

How is linearity defined when the domain is mod prime, instead of real numbers? The partial linearity argument above is stated over the reals, but Schnorr signatures operate in Zp\mathbb{Z}_pZp​. The same algebraic structure holds because Zp\mathbb{Z}_pZp​ is a field and the linearity argument only uses addition and multiplication, which are well-defined in any field.
