frost-shamir-degree-constraint

Wed Apr 01 2026

FROST uses two different methods for generating shared random numbers. Method 1 (for the shared secret sss) uses DKG, which needs two rounds of communication. Method 2 (for the nonce kkk) uses simple aggregation, which needs just one round. So can we use method 2 for generating sss, since it's faster?

No. Method 1 generates nnn Shamir secret shares that let us recreate the shared secret sss with just ttt of them. Method 2 creates nnn additive secret shares, which requires all nnn of them to recreate the shared secret kkk. The threshold property comes from Shamir sharing, and simple aggregation doesn't give you that.

Can't we just convert additive shares to Shamir shares?

Can't we use method 2 to generate nnn additive shares of sss, then later convert them to Shamir shares using the Lagrange coefficients λi\lambda_iλi​?

This doesn't solve the problem. After converting nnn additive shares to nnn Shamir shares, the underlying polynomial will have degree n−1n-1n−1, not t−1t-1t−1. So we would still need all nnn Shamir shares to reconstruct sss.

In method 1, the underlying secret polynomial has degree t−1t-1t−1, so on converting these Shamir shares to additive, we only need ttt additive shares to reconstruct the secret sss.

Signing without reconstructing sss

How is the Schnorr signature created without ever reconstructing the shared secret sss? Because the additive shares λi⋅si\lambda_i \cdot s_iλi​⋅si​ (of α\alphaα co-signers) used to generate sss are embedded inside the partial signatures ziz_izi​. When the signature aggregator combines them, it's implicitly combining these additive shares. If these were aggregated standalone, they would in fact recreate sss.
